# Phase 2 — Clients (Plan)

**Date:** 2026-06-23
**Phase:** 2 of 9 (per `docs/idea.md`)
**Depends on:** Phase 1 (Foundation & SaaS Structure) — ✅ completed & verified

## Goal

Deliver the first tenant-scoped module: **Clients** — the customer accounts at the heart of the
panel. Build it on a reusable `BelongsToCompany` trait so this and every later module (Services,
Invoices, Tickets, Domains) inherits automatic `company_id` isolation. Cover the full Phase 2
checklist from the idea doc: model + migration + factory, Flux list with search/filter/pagination,
create/edit modal, client profile page, and internal client notes.

## Decisions (confirmed with user)

- **Tenant scoping:** Build a reusable `BelongsToCompany` trait now (global scope auto-filters by the
  authenticated user's `company_id` and auto-sets it on create). Clients is the first consumer.
- **Scope:** Full module per the idea doc — Client, list, create/edit modal, profile page, ClientNote.
- **Client login-as:** Deferred. v1 is admin-only (no client portal exists to log into); revisit when
  a client portal lands.

### Database tables
- `clients`: `id`, `company_id` (FK→companies, cascade), `name`, `email`, `phone` (nullable),
  `company_name` (nullable — the client's own business name; avoids clash with tenant "company"),
  `address` (nullable), `city`/`state`/`postcode`/`country` (nullable; `country` 2-char code for filter),
  `currency` (default from settings later; default `INR` for now), `language` (default `en`),
  `status` enum (`active` | `inactive` | `closed`, default `active`), `notes` is a separate table,
  timestamps, softDeletes.
- `client_notes`: `id`, `company_id` (FK), `client_id` (FK→clients, cascade), `user_id`
  (FK→users, nullable on delete — the admin author), `body` text, timestamps.

## Implementation Steps

1. **`BelongsToCompany` trait** — `static::addGlobalScope` filtering `company_id` to
   `Auth::user()?->company_id` when a non-SaaS user is authenticated (skip scope when no auth context,
   e.g. console/seeders, or when SaaS admin — they don't query tenant data directly). `creating` hook
   sets `company_id` from the auth user if unset. Add a `company()` BelongsTo. Document What/Why/When.
2. **Migrations** — `clients` and `client_notes` per the table spec. Run `php artisan make:migration`.
3. **Models** — `Client` (uses `BelongsToCompany`, `HasFactory`, `SoftDeletes`, `LogsActivity` with
   `getActivitylogOptions`), `ClientNote` (uses `BelongsToCompany`, `HasFactory`). Relations:
   `Client hasMany ClientNote`, `Client belongsTo Company`, `ClientNote belongsTo Client/User`.
   Add `Company hasMany Client`.
4. **Factories + seeder** — `ClientFactory` (faker name/email/phone/address/country/currency/status
   states), `ClientNoteFactory`. `ClientSeeder` creates 10 clients for the demo company (set
   `company_id` explicitly since the seeder runs without auth).
5. **Policy** — `ClientPolicy` mapping each ability to the matching `clients.*` permission via
   `$user->can(...)`. Gates check permissions, never role names (idea doc authorization rule).
6. **Livewire `Clients\Index`** — Flux `flux:table` with `WithPagination`, `#[Url]` search (name/email),
   status filter, country filter, column sorting; create/edit via `flux:modal` with `#[Validate]`
   real-time validation; delete confirmation. Authorize each action against `ClientPolicy`.
7. **Livewire `Clients\Show`** — client profile: overview card (contact, status badge, currency/lang),
   placeholder sections for services/invoices/tickets (wired in later phases), and a Client Notes
   panel (list + add note form, author + timestamp). Authorize `view`/`update`.
8. **Routes + sidebar** — register routes; enable the Clients sidebar item gated on `clients.view`.
9. **Tests** (`ClientsTest`, Pest, using `companyAdmin()` helper) — see Verification.
10. **Pint** — `vendor/bin/pint --dirty --format agent`.

## Verification / Testing Plan

Feature tests (`php artisan test --compact --filter=Clients`):
- **Tenant isolation (core):** a client created under company A is invisible to a company-B admin
  (global scope). Listing as company A returns only A's clients.
- **Auto-fill:** creating a client as a company admin stamps the correct `company_id` without passing it.
- **Access control:** company admin without `clients.view` is forbidden the list; without `clients.create`
  cannot create; SaaS admin / guest blocked by existing middleware.
- **CRUD via Livewire:** create (validation: required name+email, unique email per company), edit, delete
  (soft delete), status/country filter, search.
- **Notes:** adding a note attaches author + client + company; notes are tenant-isolated.
- Run full suite afterward to confirm no regression (Phase 1 baseline: 45 passed).

Manual verification (halt and ask user per workflow): log in as demo manager → Clients list shows 10
seeded clients, create/edit/delete works, profile page + note add works, search/filter works.

## Risks & Assumptions

- **Global scope + console safety:** the scope must no-op when there's no authenticated tenant user, or
  seeders/commands would silently filter everything. Mitigation: guard on `Auth::check() && !isSaasAdmin()`
  and set `company_id` explicitly in the seeder.
- **SaaS admin access:** SaaS admins have no tenant data access except via impersonation (out of scope
  here). The scope returning empty for them is acceptable; they don't hit `/admin/clients`.
- **Unique email:** scoped uniqueness (unique per `company_id`, not globally) so two tenants can have a
  client with the same email. Enforced in validation; consider a composite unique index — will add
  `unique(['company_id','email'])` on the clients table.
- **`migrate:fresh` not required:** these are additive migrations; run `php artisan migrate` only.
- **Currency/language defaults** are hardcoded (`INR`/`en`) until Phase 8 Settings exists.
- **Activity log** on Client follows the existing Company pattern (v5 API already corrected in Phase 1).
