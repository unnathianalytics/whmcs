# Phase 2 — Clients (Completion Report)

**Date:** 2026-06-23
**Plan:** `docs/plans/2026-06-23-1831-phase-2-clients.md`
**Status:** ✅ Completed & user-verified

## Summary

Delivered the first tenant-scoped module — **Clients** — on a reusable isolation foundation:

- **`BelongsToCompany` trait** — a global scope that auto-filters tenant models by the authenticated
  company admin's `company_id`, plus a `creating` hook that auto-stamps it. No-ops for SaaS admins,
  guests and the console so seeders/maintenance are never silently filtered. Every later module
  (Services, Invoices, Tickets, Domains) will reuse it.
- **Clients module** — `Client` + `ClientNote` models, `ClientStatus` enum (Active/Inactive/Closed
  with labels + Flux badge colours), migrations with per-tenant unique email, a permission-based
  `ClientPolicy`, factories, and a `ClientSeeder` (10 demo clients).
- **UI** — Flux `Clients\Index` (table with search, status + country filters, sortable columns,
  pagination, create/edit modal, delete confirmation) and `Clients\Show` (client profile: contact
  overview, service/invoice/ticket placeholder cards, fully working internal notes panel).
- **Sidebar** Clients link enabled (gated on `clients.view`); `/admin/clients` + `/admin/clients/{client}`
  routes wired.
- **Multi-tenant RBAC fix** — replaced spatie's default team resolver with `AuthCompanyTeamResolver`
  so company-scoped permission checks resolve correctly during Livewire requests (see Deviations).

## Discovery Notes / Deviations

- **Multi-tenant RBAC bug discovered during verification (root-caused & fixed):** spatie roles/permissions
  are team-scoped (`team_id = company_id`). `EnsureCompanyAdmin` binds the team id, but it only runs on
  `/admin/*` page loads — Livewire actions post to `/livewire/update` and skip that middleware, leaving
  the team id null so every in-action permission check failed with 403 ("Create client gives 403").
  This was a latent Phase 1 gap that only surfaced now because Clients is the first feature to authorize
  inside Livewire actions. Fixed by a custom `AuthCompanyTeamResolver` (configured via
  `permission.team_resolver`) that returns an explicitly-set team id when present (seeders, onboarding,
  tests) and otherwise falls back to the authenticated company admin's `company_id`. Resolves correctly
  on page loads, Livewire updates, and any authenticated context.
- `clients` migration timestamp was lowered to `130519` (before `client_notes` at `130520`) so the FK
  target table exists first on strict databases (MySQL/Postgres).
- Added `App\Enums\ClientStatus` (first enum in the project) to centralise statuses, labels and badge
  colours — a small expansion beyond the plan's plain-string status, kept consistent with PHP rules.
- `BelongsToCompany` deliberately no-ops outside an authenticated tenant context; `ClientSeeder` sets
  `company_id` explicitly because it runs without auth.

## Follow-up Recommendations

- Phase 1 follow-up to extract per-company role seeding into an action and call it from
  `Saas\Companies\Index::createCompany` is still open — new tenants created via the UI still get no roles
  until seeded. Worth closing before tenants self-serve.

  `EnsureCompanyAdmin` is redundant for permission resolution (harmless; kept as a defensive belt-and-braces).
  Consider documenting that the resolver — not the middleware — is what guarantees tenant-scoped checks.
- As later tenant models are added, have each `use BelongsToCompany` and add a matching isolation test
  mirroring `ClientsTest`'s "tenant isolation" group.
